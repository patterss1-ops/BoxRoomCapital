## Context
Ticket `A-002` delivers a unified order intent envelope that standardizes execution lifecycle persistence across brokers while preserving compatibility with the existing `order_actions` state machine.

## Changes
- Added `execution/order_intent.py`:
  - `OrderIntent` model with required fields (`strategy_id`, `strategy_version`, `sleeve`, `account_type`, `broker_target`, `instrument`, `side`, `qty`, `order_type`, `risk_tags`).
  - Canonical enums for side/type/status.
  - Actor normalization (`system`/`operator`) for audit records.
- Added `data/order_intent_store.py`:
  - Schema bootstrap for:
    - `order_intents`
    - `order_intent_attempts`
    - `order_intent_transitions`
  - `create_order_intent_envelope(...)` to create stable `intent_id` + `correlation_id` and a linked `order_actions` row.
  - `transition_order_intent(...)` to persist status transitions with actor/timestamps and append retry attempts (upsert-by-attempt, no overwrite of prior attempts).
  - Query helpers for intents, attempts, and transitions.
  - Payload snapshot persistence for both request and response envelopes.
- Added tests:
  - `tests/test_order_intent_model.py`
  - `tests/test_order_intent_lifecycle.py`

## Risks
- Existing runtime paths (e.g. `options_runner.py`) are not yet using the new intent store; A-002 provides the contract + persistence layer and compatibility bridge, but full production-path wiring is still pending.
- Transition rules are strict and may need expansion once IBKR path semantics are finalized in A-003/A-004.

## Next Action
- Wire live/manual execution entry points to call `create_order_intent_envelope` and `transition_order_intent` so all broker submissions emit the new envelope by default.
- Use this envelope as the input contract for routing (`A-004`) and ledger attribution (`A-005`).
