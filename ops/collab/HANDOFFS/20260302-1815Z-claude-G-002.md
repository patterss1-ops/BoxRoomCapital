## Summary

Built the execution quality analytics layer that consumes G-001 telemetry
(order_execution_metrics) and produces fill-rate, slippage, latency, and
per-broker/strategy breakdowns for operator dashboard and G-004 AI
confidence gate calibration.

## Changes

### `fund/execution_quality.py` (NEW — 340 lines)

Core analytics module with 7 dataclasses and 10 functions:

| Component | Description |
|-----------|-------------|
| `FillStats` | Fill rate, reject rate, partial fill rate, avg fill ratio |
| `SlippageStats` | Mean/median/P5/P95/max/min bps + total slippage cost |
| `LatencyStats` | Mean/P50/P95/max dispatch latency in ms |
| `BrokerBreakdown` | Per-broker fill rate, reject rate, mean slippage, mean latency |
| `StrategyBreakdown` | Per-strategy fill rate, reject rate, mean slippage, notional traded |
| `ExecutionQualityReport` | Complete windowed report with verdict |
| `compute_fill_stats()` | Fill/reject/partial analytics from metrics rows |
| `compute_slippage_stats()` | Slippage distribution with percentiles |
| `compute_latency_stats()` | Latency distribution with percentiles |
| `compute_broker_breakdown()` | Group metrics by broker |
| `compute_strategy_breakdown()` | Group metrics by strategy |
| `_compute_verdict()` | healthy/attention/degraded/no_data classification |
| `build_execution_quality_report()` | Full report builder with time window filtering |
| `get_execution_quality_payload()` | JSON-safe dict for API/UI consumption |

Verdict thresholds:
- **healthy**: fill rate >= 90% AND mean slippage <= 20 bps
- **attention**: fill rate >= 70% AND mean slippage <= 50 bps
- **degraded**: below attention thresholds
- **no_data**: zero attempts in window

### `app/api/server.py` (MODIFIED — 2 endpoints added)

- `GET /api/execution-quality?days=30` — JSON payload with all analytics
- `GET /fragments/execution-quality` — HTML fragment for operator dashboard

### `app/web/templates/_execution_quality.html` (NEW)

Operator dashboard fragment with:

- Fill metrics table (total, completed, failed, retrying, rates)

- Latency distribution table (mean/P50/P95/max)
- Per-broker breakdown table
- Per-strategy breakdown table
- Graceful empty state for no_data

- Latency stats include ALL statuses (not just completed) — measures dispatch infrastructure
- Verdict thresholds are tunable; current values match spec expectations
- `_percentile()` uses interpolated nearest-rank (matches numpy behavior)
